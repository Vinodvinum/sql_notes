Advanced Foundation

Deeper schema understanding: data types, keys, constraints, and CAST

These topics are often skipped by beginners, but they matter in interviews because they show whether you understand how tables are really designed.

Simple idea: good SQL is not only writing queries. It is also understanding what kind of data is stored, how tables connect, and what rules protect data quality.
Part 1

Data Types (deeper understanding)

Simple meaning

Data type means the kind of data stored in a column.

Column Data type Meaning
customer_idINTwhole numbers
nameVARCHAR(50)text up to 50 characters
order_dateDATEdate only
salaryDECIMAL(10,2)numeric value with 2 decimal places
statusENUM(...)fixed set of allowed values

Why it matters

If salary is a number column, writing text like 'fifty thousand' is wrong. The database needs the correct type to store and compare values properly.

Interview answer

Data types define what kind of values a column can store, such as numbers, text, dates, or decimals. They are important for data accuracy, validation, and correct query behavior.

Part 2

Primary Key, Foreign Key, and Constraints

Concept Simple meaning
Primary Keyuniquely identifies each row
Foreign Keyconnects one table to another
NOT NULLvalue cannot be empty
UNIQUEduplicate values are not allowed
DEFAULTautomatic value if none is given
 
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE,
    city VARCHAR(50)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    amount DECIMAL(10,2),
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

Interview answer

Primary key uniquely identifies each row, while foreign key creates a relationship between tables. Constraints like NOT NULL and UNIQUE help maintain data quality.

Part 3

ENUM and complete create table thinking

Simple meaning

ENUM is used when a column should allow only a small fixed set of values.

CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    city VARCHAR(50),
    salary DECIMAL(10,2),
    join_date DATE,
    status ENUM('active', 'inactive')
);

Where it helps

ENUM is useful for status columns like pending, shipped, delivered, active, inactive, approved, rejected.

Part 4

CAST: type conversion

Simple meaning

CAST changes one data type into another type when needed.

SELECT CAST(salary AS FLOAT)
FROM employees;
SELECT CAST(order_date AS DATE)
FROM orders;

Interview answer

CAST is used for type conversion when data types do not match or when a specific data type is needed for calculation, formatting, or comparison.

Extra Join Types

RIGHT JOIN and FULL OUTER JOIN idea

These are not always used daily in analyst roles, but understanding them helps in interviews and deeper SQL thinking.

Join Extra 1

RIGHT JOIN

Simple meaning

RIGHT JOIN keeps all rows from the right table and matching rows from the left table.

SELECT c.name, o.amount
FROM customers c
RIGHT JOIN orders o
ON c.customer_id = o.customer_id;

Think like this

If the right table is orders, every order stays in the output even if matching customer data is missing.

Interview answer

RIGHT JOIN returns all records from the right table and matching records from the left table. If no match exists, the left-side columns become NULL.

Join Extra 2

FULL OUTER JOIN

Simple meaning

FULL OUTER JOIN keeps all rows from both tables. Matching rows combine, non-matching rows still stay with NULL on the missing side.

SELECT c.name, o.amount
FROM customers c
FULL OUTER JOIN orders o
ON c.customer_id = o.customer_id;

Important note

Some databases support FULL OUTER JOIN directly. MySQL usually needs a workaround using UNION of LEFT JOIN and RIGHT JOIN logic.

Interview answer

FULL OUTER JOIN returns all records from both tables, whether matching exists or not. It is useful when we want complete visibility from both sides.

Window Deep

Window functions complete understanding with output thinking

This part continues the window function topic more deeply, exactly like the document does after saying the first explanation was not enough.

Key idea: GROUP BY collapses rows. Window functions keep rows and add extra intelligence.
Window Add-on 1

RANK vs DENSE_RANK vs ROW_NUMBER

Salary RANK DENSE_RANK ROW_NUMBER
60000111
50000222
50000223
40000434

Simple meaning

RANK skips numbers after ties.

DENSE_RANK does not skip numbers.

ROW_NUMBER always gives unique serial numbers.

Strong interview answer

RANK gives the same rank to duplicate values and skips the next rank, DENSE_RANK does not skip ranks, and ROW_NUMBER gives unique sequential numbers regardless of duplicates.

Window Add-on 2

PARTITION BY deep clarity

Simple meaning

PARTITION BY means divide the data into groups, then apply the window function separately inside each group.

RANK() OVER (PARTITION BY city ORDER BY salary DESC)

Think like this

Instead of one big table, you temporarily make many small windows. Then ranking or running total happens inside each window.

Window Add-on 3

Group-wise ranking with output

name city salary
VinodMysore30000
AshaMysore25000
RaviBangalore50000
MeenaBangalore40000
KiranBangalore60000
 
SELECT name, city, salary,
RANK() OVER (PARTITION BY city ORDER BY salary DESC) AS rank
FROM employees;

Output

name city salary rank
VinodMysore300001
AshaMysore250002
KiranBangalore600001
RaviBangalore500002
MeenaBangalore400003

How to think

First divide by city. Then inside each city, sort by salary descending. Then ranking resets for each city.

Window Add-on 4

Running total with output

order_id amount
11000
22000
31500
43000
 
SELECT order_id, amount,
SUM(amount) OVER (ORDER BY order_id) AS running_total
FROM orders;

Output

order_id amount running_total

Interview answer

Running total is a cumulative sum calculated row by row using a window function, where each row includes the sum of previous rows based on a defined order.

Mock Interview

Mock interview round 1 with ready-to-speak scripts

This section follows the document’s interview style closely: not just SQL syntax, but how to speak the logic clearly.

Important speaking pattern: First I understand ... Then I use ... Finally I filter or sort ...
Round 1

WHERE vs HAVING

Ready script

WHERE is used to filter rows before grouping happens, while HAVING is used to filter aggregated results after GROUP BY. For example, if I want employees from Bangalore, I use WHERE. But if I want cities where total salary is greater than 50000, I use HAVING because aggregation is involved.

Round 1

Total revenue per customer

Thinking script

First, I understand that revenue comes from the orders table, but customer names come from the customers table. So I need to join both tables using customer_id. Then I aggregate total revenue using SUM and group by customer.

SELECT c.name, SUM(o.amount) AS total_revenue
FROM customers c
INNER JOIN orders o
ON c.customer_id = o.customer_id
GROUP BY c.name;
Round 1

Customers with no orders

Thinking script

To find customers with no orders, I use LEFT JOIN because I want all customers even if they do not have matching orders. Then I filter rows where order data is NULL, which indicates no orders.

SELECT c.name
FROM customers c
LEFT JOIN orders o
ON c.customer_id = o.customer_id
WHERE o.customer_id IS NULL;
Round 1

Second highest salary

Thinking script

To find the second highest salary, I use a window function like RANK to rank salaries in descending order, and then I filter where rank equals 2.

SELECT name, salary
FROM (
    SELECT name, salary,
           RANK() OVER (ORDER BY salary DESC) AS rnk
    FROM employees
) t
WHERE rnk = 2;
Round 1

Rank employees within each city

Thinking script

Since ranking needs to be done within each city, I use PARTITION BY city to divide data into groups, and then ORDER BY salary descending to assign ranks within each group.

SELECT name, city, salary,
RANK() OVER (PARTITION BY city ORDER BY salary DESC) AS rank
FROM employees;
Round 1

Unknown SQL problem

Ready script

First, I try to understand the business requirement clearly. Then I break it into smaller steps like identifying required tables, joins, and aggregations. Even if I do not know exact syntax, I focus on logic and build the query step by step.

Round 2

Harder questions and stronger answer scripts

This is closer to actual advanced rounds where interviewers want both logic and speaking clarity.

Advanced Q1

Top 2 customers per city

Ready script

First, I understand that ranking should be done within each city, not globally. So I use a window function with PARTITION BY city to separate data by city. Then I order by revenue in descending order and assign ranks. Finally, I filter for top 2 ranks.

SELECT name, city, total_spent
FROM (
    SELECT c.name, c.city, SUM(o.amount) AS total_spent,
           RANK() OVER (PARTITION BY c.city ORDER BY SUM(o.amount) DESC) AS rnk
    FROM customers c
    INNER JOIN orders o
    ON c.customer_id = o.customer_id
    GROUP BY c.name, c.city
) t
WHERE rnk <= 2;
Advanced Q2

Duplicate records by email

Ready script

To find duplicate emails, I group the data by email and use COUNT. Then I keep only those groups where count is greater than 1 using HAVING.

SELECT email, COUNT(*)
FROM customers
GROUP BY email
HAVING COUNT(*) > 1;
Advanced Q3

When interviewer says: what if there are ties?

Ready script

If ties matter, I would use RANK or DENSE_RANK instead of LIMIT so that tied records are handled properly. LIMIT is useful for a fixed number of rows, but ranking is better when equal values need fair treatment.

Advanced Q4

Why not use GROUP BY for ranking?

Ready script

GROUP BY reduces rows and gives aggregated output, whereas ranking needs row-level visibility. Window functions keep all rows and add rank information without collapsing the data.

Top 10 Questions

Top 10 Data Analyst SQL questions with input, SQL, output, and explanation

If you understand these 10 patterns, you can handle a large part of data analyst SQL interviews.

Question 1

Top N customers

customer_idamount
15000
13000
27000
32000
 
SELECT customer_id, SUM(amount) AS total
FROM orders
GROUP BY customer_id
ORDER BY total DESC
LIMIT 2;

Output: customer 1 -> 8000, customer 2 -> 7000

Add amounts per customer, sort descending, then take top 2.

Question 2

Revenue per category

orders.product_idorders.amountproducts.category
15000Electronics
23000Clothing
 
SELECT p.category, SUM(o.amount) AS revenue
FROM orders o
JOIN products p ON o.product_id = p.product_id
GROUP BY p.category;

Join product category, then group and sum revenue.

Question 3

Customers with no orders

SELECT c.name
FROM customers c
LEFT JOIN orders o
ON c.id = o.customer_id
WHERE o.customer_id IS NULL;

LEFT JOIN plus NULL check is the classic missing-data pattern.

Question 4

Average order value

SELECT AVG(amount) FROM orders;

Use AVG when the question asks average, mean, or typical value.

Question 5

Monthly sales trend

SELECT MONTH(order_date) AS month, SUM(amount)
FROM orders
GROUP BY MONTH(order_date);

Extract month from date, group by month, then sum revenue.

Question 6

Second highest salary

SELECT MAX(salary)
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

Remove the highest salary first, then take the maximum from what remains.

Question 7

Duplicate records

SELECT email, COUNT(*)
FROM customers
GROUP BY email
HAVING COUNT(*) > 1;

Duplicate detection is usually GROUP BY plus HAVING COUNT greater than 1.

Question 8

Running total

SELECT order_id, amount,
SUM(amount) OVER (ORDER BY order_id) AS running_total
FROM orders;

Each row adds previous values, so the total keeps growing row by row.

Question 9

Rank per group

SELECT name, city, salary,
RANK() OVER (PARTITION BY city ORDER BY salary DESC) AS rank
FROM employees;

Ranking resets for each city because PARTITION BY creates separate groups.

Question 10

Repeat customers

SELECT customer_id
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > 1;

If count is more than 1, that customer placed orders more than once.

Case Study

Real Amazon-style case study answer

This is the type of answer that sounds much stronger than only giving syntax.

Scenario

Sales dropped last month. How will you analyze?

Perfect structured answer

First, I will understand what exactly dropped, whether it is total revenue, number of orders, or average order value. Then I will break the problem into customer activity, number of orders, revenue per order, and category performance. After that, I will use SQL to check trends over time, category-wise revenue, and customer behavior. Finally, I will summarize the likely reasons and suggest actions based on the findings.

-- Total sales trend
SELECT MONTH(order_date), SUM(amount)
FROM orders
GROUP BY MONTH(order_date);

-- Orders count
SELECT COUNT(*) FROM orders;

-- Category performance
SELECT p.category, SUM(o.amount)
FROM orders o
JOIN products p ON o.product_id = p.product_id
GROUP BY p.category;

-- Customer activity
SELECT customer_id, COUNT(*)
FROM orders
GROUP BY customer_id;

Possible insights

Sales may drop because active customers decreased, high-value orders reduced, important categories underperformed, or there were seasonal factors.

Final Revision

One-page final revision and final interview simulation

This is the fast last-round revision from the document, but arranged neatly in HTML.

Core flow

SELECT -> FROM -> JOIN -> WHERE -> GROUP BY -> HAVING -> ORDER BY -> LIMIT

Golden line

First I understand the requirement, then identify tables, apply joins, use aggregation if needed, and finally filter and sort the results.

Most important patterns

  • Top N -> ORDER BY + LIMIT
  • Per group -> GROUP BY
  • Missing data -> LEFT JOIN + NULL
  • Ranking -> RANK()
  • Running total -> SUM() OVER()
