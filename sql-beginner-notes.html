01 — The Basics

What is SQL? (In Plain English)

Imagine SQL as a way to ask questions to a giant spreadsheet that has millions of rows. That's literally it.

🧠 Think of it this way
You work at a company. All customer data is stored in a database (think: a collection of Excel files). You want to know: "How many customers bought something last month?"

Instead of manually scrolling through 1 million rows, you write a SQL question and the database gives you the answer in seconds. That question is called a query.

Database = Collection of Tables

02 — Command #1

SELECT — "Show me this data"

SELECT is the most used SQL command. It means: "Show me these columns from this table." You write it first, but think of it as: what do I want to see?

🧠 Real life analogy
Imagine you have a big filing cabinet with customer cards. SELECT is you saying "I only want to see the Name and City columns" — not every single detail.
SQLYour very first query — reading it like English
-- The basic structure: SELECT what  FROM where

SELECT name, city          -- "Show me name and city columns"
FROM customers;            -- "from the customers table"

-- Result:
-- name          city
-- Arjun Sharma  Delhi
-- Priya Nair    Mumbai
-- Meera Patel   Delhi
SQLMore SELECT examples
-- Show ALL columns (use * = "everything")
SELECT *
FROM customers;

-- Rename a column in the output using AS
SELECT
    name         AS customer_name,    -- rename it
    city         AS location,
    amount       AS order_value
FROM orders;

-- Do math in SELECT (create new columns)
SELECT
    order_id,
    amount,
    amount * 0.18     AS tax_amount,     -- 18% tax
    amount * 1.18     AS total_with_tax  -- total including tax
FROM orders;
💡 Simple Rule: Every SQL query starts with SELECT. Think of it as telling the database what you want to SEE. The ; at the end is just like a full stop — it tells SQL your sentence is done.

DISTINCT — Show Unique Values Only

SQLRemove duplicates from results
-- Without DISTINCT: shows Delhi, Mumbai, Delhi (Delhi repeats!)
SELECT city FROM customers;

-- With DISTINCT: shows Delhi, Mumbai (no repeats)
SELECT DISTINCT city FROM customers;

-- Real use: "What unique products do we sell?"
SELECT DISTINCT product_name FROM products;
03 — Command #2

WHERE — "But only show rows that match..."

WHERE filters your data. It's like adding a condition: "Show me customers, BUT ONLY the ones from Delhi." WHERE is the second most important command after SELECT.

🧠 Think of it this way
You're searching through 50,000 customer records. WHERE is your search filter — like filtering a spreadsheet. Instead of scrolling through everything, you get only the rows that match your condition.
SQLWHERE with all comparison types
-- Equal to
SELECT * FROM customers WHERE city = 'Delhi';

-- NOT equal to
SELECT * FROM orders WHERE status != 'Cancelled';

-- Greater than / Less than
SELECT * FROM orders WHERE amount > 1000;
SELECT * FROM orders WHERE amount < 500;
SELECT * FROM orders WHERE amount >= 1000;  -- 1000 or more

-- Date filtering
SELECT * FROM orders WHERE order_date >= '2024-01-01';

-- IN: instead of writing OR OR OR
SELECT * FROM customers
WHERE city IN ('Delhi', 'Mumbai', 'Bangalore');

-- BETWEEN: range of values (includes both ends)
SELECT * FROM orders
WHERE amount BETWEEN 500 AND 2000;

-- LIKE: search inside text (% means "anything")
SELECT * FROM customers WHERE name LIKE 'A%';      -- starts with A
SELECT * FROM customers WHERE email LIKE '%gmail%'; -- contains "gmail"

AND / OR — Multiple Conditions

SQLCombining conditions
-- AND: BOTH conditions must be true
SELECT * FROM orders
WHERE amount > 1000
  AND status = 'Delivered';
-- Only shows: big orders that were actually delivered

-- OR: EITHER condition is true
SELECT * FROM customers
WHERE city = 'Delhi'
   OR city = 'Mumbai';
-- Shows customers from Delhi OR Mumbai

-- IS NULL: find missing/empty values
SELECT * FROM customers WHERE phone IS NULL;
-- Shows customers who haven't given their phone number
⚠️ Common Mistake! To check for empty/missing values, you MUST write IS NULL — NEVER = NULL. Writing WHERE phone = NULL will always return zero results. SQL has this quirk — just remember it!
04 — Commands #3 & #4

ORDER BY & LIMIT — Sort & Top N

ORDER BY sorts your results. LIMIT tells SQL "stop after N rows." Together they give you Top-N lists — one of the most common analyst queries.

SQLSorting results + getting top N
-- Sort by amount: highest first
SELECT order_id, customer_id, amount
FROM orders
ORDER BY amount DESC;   -- DESC = largest first

-- Sort: smallest first
SELECT * FROM orders
ORDER BY amount ASC;    -- ASC = smallest first (default)

-- Top 10 biggest orders
SELECT *
FROM orders
ORDER BY amount DESC
LIMIT 10;             -- Stop after 10 rows

-- Sort by two columns: city alphabetically, then amount biggest first
SELECT name, city, amount
FROM orders
ORDER BY city ASC, amount DESC;
💡 Memory trick: ORDER BY is always at the END of your query. LIMIT comes last of all. Think: "Show me [data] from [table] where [condition] sorted [by what] and only [N rows]."
05 — Commands #5, 6, 7

GROUP BY, COUNT, SUM — "Calculate per category"

This is where SQL gets really powerful. GROUP BY lets you calculate totals, averages, and counts FOR EACH category — like "total sales per city" or "number of orders per customer."

🧠 Think of it this way
You have 50,000 orders from all over India. You want to know: "What is the total revenue from each city?"

GROUP BY groups all Delhi orders together, all Mumbai orders together, etc. Then it runs a calculation (like SUM) on each group separately.

The 5 Calculation Functions

COUNT(*)
Count how many rows are in the group
COUNT(*) → 1250 orders
SUM(col)
Add up all values in a column
SUM(amount) → ₹5,24,000
AVG(col)
Calculate the average value
AVG(amount) → ₹418.4
MAX(col)
Find the highest value
MAX(amount) → ₹9,200
MIN(col)
Find the lowest value
MIN(amount) → ₹99
SQLGROUP BY examples — read them like sentences
-- "Count how many orders came from each city"
SELECT
    city,
    COUNT(*) AS number_of_orders
FROM orders
GROUP BY city;

-- Result:
-- city      number_of_orders
-- Delhi     450
-- Mumbai    320
-- Bangalore 280

-- "Total revenue from each city, sorted biggest first"
SELECT
    city,
    COUNT(*) AS orders,
    SUM(amount) AS total_revenue,
    AVG(amount) AS avg_order_size
FROM orders
GROUP BY city
ORDER BY total_revenue DESC;

HAVING — Filter AFTER Grouping

WHERE filters rows BEFORE grouping. HAVING filters AFTER grouping. Use HAVING when you want to filter groups based on a calculated value.

WHERE — filters ROWS first
Use when filtering by a normal column value.

WHERE city = 'Delhi' ✓
WHERE status = 'Delivered' ✓

Cannot use SUM, COUNT etc. here.
HAVING — filters GROUPS after
Use when filtering by a calculated value.

HAVING SUM(amount) > 50000 ✓
HAVING COUNT(*) >= 10 ✓

Use SUM, COUNT, AVG here.
SQLWHERE and HAVING used together
-- "Cities with more than ₹50,000 in DELIVERED orders"
SELECT
    city,
    COUNT(*) AS orders,
    SUM(amount) AS revenue
FROM orders
WHERE status = 'Delivered'     -- filter rows first
GROUP BY city
HAVING SUM(amount) > 50000   -- then filter groups
ORDER BY revenue DESC;
🔑 Golden Rule: Every column in SELECT must either be in GROUP BY, OR inside a function like SUM/COUNT. If you SELECT a column that's not in GROUP BY and not in a function, SQL will give you an error.
06 — Command #8

JOIN — "Combine two tables"

JOIN is how you connect two tables together. Real data always lives in multiple tables, and JOIN is how you bring them together in one query. This is one of the most important SQL skills.

🧠 Think of it this way
You have two spreadsheets: one with customer info (name, city) and one with orders (what they bought, how much). They're connected by customer_id.

JOIN merges them into one table so you can say: "Show me Arjun's name AND his orders."

3 Types of JOIN

INNER JOIN
Only rows that match in BOTH tables. Most common type.
LEFT JOIN
ALL left table rows. Right table adds info where it can. NULLs where there's no match.
FULL OUTER JOIN
ALL rows from both tables. NULLs wherever there's no match on either side.
SQLJOIN examples — step by step
-- INNER JOIN: show customer name + their orders (only customers who ordered)
SELECT
    c.name,          -- from customers table
    c.city,          -- from customers table
    o.order_id,      -- from orders table
    o.amount         -- from orders table
FROM customers c                           -- "c" is a shortcut for customers
INNER JOIN orders o ON c.cust_id = o.cust_id; -- connect where IDs match

-- LEFT JOIN: show ALL customers, even those with no orders
SELECT
    c.name,
    o.order_id,   -- will be NULL for customers with no orders
    o.amount      -- will be NULL for customers with no orders
FROM customers c
LEFT JOIN orders o ON c.cust_id = o.cust_id;

-- TRICK: Find customers who NEVER placed an order
SELECT c.name, c.email
FROM customers c
LEFT JOIN orders o ON c.cust_id = o.cust_id
WHERE o.order_id IS NULL;   -- if order_id is NULL, they never ordered!
💡 Simple way to remember: When you JOIN tables, use short aliases (like c for customers, o for orders) so you can write c.name instead of customers.name. Always write the joining condition after ON — it's always the matching ID column between the two tables.
07 — Command #9

CASE WHEN — "If this, then that"

CASE WHEN is SQL's way of making decisions. It lets you create a new column based on conditions — like "If amount is more than 5000, call it Big Order. Otherwise, call it Small Order."

SQLCASE WHEN — like if/else in SQL
-- Classify orders by size
SELECT
    order_id,
    amount,
    CASE
        WHEN amount >= 10000 THEN 'Big Order'
        WHEN amount >= 2000  THEN 'Medium Order'
        WHEN amount >= 500   THEN 'Small Order'
        ELSE 'Tiny Order'       -- if none of the above match
    END AS order_size           -- name the new column
FROM orders;

-- Classify customers by loyalty level
SELECT
    cust_id,
    SUM(amount) AS total_spent,
    CASE
        WHEN SUM(amount) >= 50000 THEN 'Platinum'
        WHEN SUM(amount) >= 20000 THEN 'Gold'
        WHEN SUM(amount) >= 5000  THEN 'Silver'
        ELSE 'Bronze'
    END AS tier
FROM orders
GROUP BY cust_id;
08 — Command #10

Window Functions — "Calculate without losing rows"

This is slightly advanced but very important for jobs. Window functions do calculations like SUM and RANK — BUT they keep every row in the result instead of collapsing them like GROUP BY.

🧠 GROUP BY vs Window — the key difference
GROUP BY: "Give me total sales per city" → You get 4 rows (one per city). Original data is gone.

Window Function: "Add the city total NEXT TO each row" → You keep ALL 50,000 rows AND each row shows its city's total beside it.
GROUP BY — collapses rows
city     | total
———————————
Delhi   | 38,500
Mumbai | 12,000

Only 2 rows. Original data gone!
Window — keeps all rows
name  | amt  | city_total
————————————————
Arjun | 4500 | 38,500
Arjun | 800  | 38,500
Rahul | 1800 | 38,500

All rows kept! Total added as column.
SQLThe 3 most important window functions for a job
-- 1. ROW_NUMBER: Give each row a rank number
SELECT
    name,
    city,
    amount,
    ROW_NUMBER() OVER (ORDER BY amount DESC) AS rank_num
FROM orders;
-- rank_num: 1, 2, 3, 4... biggest orders first

-- 2. RANK per group (Top customer in EACH CITY)
SELECT
    name, city, total_spend,
    ROW_NUMBER() OVER (
        PARTITION BY city            -- restart ranking for each city
        ORDER BY total_spend DESC    -- highest spender = rank 1
    ) AS rank_in_city
FROM customer_totals;

-- 3. Running Total (cumulative sum month by month)
SELECT
    month,
    monthly_revenue,
    SUM(monthly_revenue) OVER (
        ORDER BY month
    ) AS running_total         -- grows each month
FROM monthly_summary;

-- Classic Interview Pattern: Top 1 customer per city
WITH ranked AS (
    SELECT name, city, total_spend,
        ROW_NUMBER() OVER (PARTITION BY city ORDER BY total_spend DESC) AS rn
    FROM customer_totals
)
SELECT name, city, total_spend
FROM ranked
WHERE rn = 1;    -- only the #1 person in each city
ℹ️ Don't panic about window functions yet! Focus on SELECT, WHERE, GROUP BY, and JOINs first. Come back to window functions once you're comfortable. They take a bit of practice but they will "click" once you understand what problem they solve.
09 — Memory & Tips

Memory Tricks & Common Mistakes

These simple tricks will help you remember SQL faster and avoid the mistakes that even experienced people make.

The Query Writing Order — Remember This!

SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT

🧠 Mnemonic: "Some Fish Jump When Grouped, Having Order Limits"

6 Memory Tricks

WHERE = rows, HAVING = groups
WHERE filters rows BEFORE grouping. HAVING filters groups AFTER. If your condition uses SUM or COUNT → use HAVING. Otherwise → use WHERE.
LEFT JOIN + IS NULL = "never did it"
LEFT JOIN two tables, then WHERE right_id IS NULL means "rows in left table with NO match in right table." Used for: customers who never ordered, products never sold.
GROUP BY = "per category"
Whenever you want to calculate something PER something else (per city, per month, per product), you need GROUP BY.
Alias tables with one letter
Write FROM customers c and JOIN orders o. Then use c.name and o.amount. Short, clean, professional.
CTE = break it into steps
If a query feels too complex, use WITH to name each step. Each CTE = one logical step. Makes the query easy to read and debug.
NULL is not zero or empty
NULL means "unknown." You can't compare it with =. Always use IS NULL or IS NOT NULL. SUM and AVG automatically ignore NULLs.

Top 5 Beginner Mistakes

  • 1
    Writing = NULL instead of IS NULL
    Always write WHERE phone IS NULL never WHERE phone = NULL. The second one never works.
  • 2
    Forgetting to put text in quotes
    Text values need single quotes: WHERE city = 'Delhi'. Numbers don't: WHERE amount > 1000.
  • 3
    Using WHERE when you need HAVING
    If you're filtering after a GROUP BY using SUM/COUNT, you need HAVING, not WHERE.
  • 4
    Selecting a column that's not in GROUP BY
    Every column in SELECT must be in GROUP BY OR inside SUM/COUNT/AVG. Otherwise you'll get an error.
  • 5
    Using SELECT * in every query
    In real jobs, always specify which columns you need. SELECT * is slow on large tables and shows too much data.
